**v0.3.0 adds opt-in Outlook support, four Outlook-specific hardenings, two new tray menu items, and four extra `doctor.bat` checks.** Users who only run Word, Excel, or PowerPoint can upgrade safely: Outlook is fully opt-in and nothing in the existing three-app flow changes. Users who want RTL in the classic Outlook add-in panel are the primary audience.

## What's new

- Outlook support in classic Outlook on Microsoft 365, gated by a per-launch opt-in flag and a warning dialog. New Outlook (`olk.exe`) is intentionally out of scope.
- Four Outlook-specific hardenings: per-app `BLOCKED_HOST_INFO_KEYS` filter (Outlook permanently on the block list unless the opt-in flag exists), 15-minute auto-disconnect timer for any Outlook CDP attachment, URL redaction that strips every query parameter except `_host_Info=` from Outlook log entries, and a race guard in `attach()` that aborts an in-flight Outlook attach if a disconnect arrives mid-handshake.
- Tray menu gains **Connect Outlook** (warning dialog with Cancel as default-focused button, names the exact exposure rather than hand-waving) and **Disconnect Outlook only** (drops the Outlook CDP attachment without touching Word/Excel/PowerPoint or closing Outlook itself).
- `doctor.bat` extended from 15 to 19 checks: Outlook installed, Outlook running, Outlook CDP target via dynamic port discovery, Outlook entry in `apps.json`. All four are informational, since Outlook is opt-in.

## Security

The whole Outlook track was driven by an M0 finding, not a feature request. The shipped v0.2.2 injector was observed attaching to the Outlook CDP target automatically the moment the WebView2 debug flag was set on Outlook, with no consent dialog and no opt-in step, because the v0.2.x code only filtered on `pivot.claude.ai` and never on the host Office app. Shipping a Connect Outlook item without the per-app block list and per-launch opt-in would have meant v0.3.0 silently enabled CDP attach to Outlook for every user who upgraded. The opt-in model and the per-app filter in `lib/office-apps.js` close that exposure before release. No new EDR triggers introduced; the per-process environment-variable model from v0.2.0 is preserved unchanged.

## Upgrade

1. Download the v0.3.0 zip from the Releases page and extract it over your existing install folder.
2. Close Word, Excel, and PowerPoint (and Outlook, if running).
3. Run `install.bat`. Existing tray and injector are stopped via their PID files and the new build takes effect immediately.

## If you do NOT use Outlook

Nothing changes for you. Outlook is opt-in at both the tray and the injector: the new `outlook-wrapper.bat` is never invoked, the per-launch opt-in flag is never written, and Word/Excel/PowerPoint behaviour is byte-identical to v0.2.2. The injector still discovers the Outlook CDP target on each tick and logs it as blocked, which is visible in the diagnostic log but has no other effect. `apps.json` gets a fourth `Outlook` key that older tray builds simply ignore.

## Known limitations

- The Outlook target filter is a lowercased equality on the first `$`-delimited segment of `_host_Info=`. A local process that stands up its own WebView2 with a forged `_host_Info=Outlook$...` would satisfy the filter; defence against that scenario relies on the opt-in flag and the 15-minute auto-disconnect, not on the filter alone.
- The DOM mutation from a successful `Runtime.evaluate` (`<style id="__claude_rtl_fix__">`, `dir="rtl"`) is not torn down when the CDP client is closed. This is consistent with every other disconnect path (Disconnect all, app exit, navigation, auto-disconnect): none of them remove the injected style, only the CDP attachment.
